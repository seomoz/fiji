Custom Importers

In this section of the tutorial, we will import metadata about songs into the Fiji table songs, and import data about when users have listened to songs into the Fiji table users. The importer for song metadata is in the file SongMetadataImporter.scala, while the importer for a user’s song plays history is in the file SongPlaysImporter.scala. Both of these files define importers that can be run as precompiled jobs. They also exist in script form as SongMetadataImporter.express and SongPlaysImporter.express. Later, we’ll demonstrate how to run the importers both as precompiled jobs and as scripts (although it’s only necessary to run the importers one of these ways).

In this section, we’ll provide more details on the song metadata importer. The importer for a user’s song play history is similar. The importer for song metadata reads a file in HDFS that contains a JSON record of song metadata on each line, and writes metadata (contained in an Avro record) to a Fiji table.

Each JSON record contained in the input file corresponds to a song, and provides the following metadata for the song:

  • song id
  • song name
  • artist name
  • album name
  • genre
  • tempo
  • duration

Our importer will flatten each of these JSON records into Scalding tuple fields, and then pack those tuple fields into an Avro record. To do this, we’ll use a few user defined functions (UDF). A UDF in FijiExpress is simply a Scala function; any Scala function can potentially be used as a UDF in a FijiExpress flow. We’ll first explain the UDFs used in our import flow before explaining the flow itself.

The UDF parseJson takes as an argument a string containing the JSON for an individual record, and returns a Scala tuple containing one element for each bit of metadata. The code for this function is below.

def parseJson(json: String): (String, String, String, String, String, Long, Long) = {
  val metadata = JSON.parseFull(json).get.asInstanceOf[Map[String, Any]]
  (metadata.get("song_id").get.asInstanceOf[String],
      metadata.get("song_name").get.asInstanceOf[String],
      metadata.get("album_name").get.asInstanceOf[String],
      metadata.get("artist_name").get.asInstanceOf[String],
      metadata.get("genre").get.asInstanceOf[String],
      metadata.get("tempo").get.asInstanceOf[String].toLong,
      metadata.get("duration").get.asInstanceOf[String].toLong)
}

As an example, suppose this function received the following JSON string as input:

{ "song_id" : "song-0", "song_name" : "song name-0", "artist_name" : "artist-1", "album_name" : "album-1", "genre" : "genre5.0", "tempo" : "100", "duration" : "240" }

It would then return the following Scala tuple:

(“song-0”, "song name-0", "artist-1", "album-1", "genre5.0", 100, 240)

Using this UDF, we can write our import flow.

TextLine(args("input"))
    .map('line ->
        ('songId, 'songName, 'albumName, 'artistName, 'genre, 'tempo,'duration)) { parseJson }
    .map('songId -> 'entityId) { songId: String => EntityId(args("table-uri"))(songId) }
    .pack[SongMetadata](('songName, 'albumName, 'artistName, 'genre, 'tempo, 'duration)
        -> 'metadata)
    .write(FijiOutput(args("table-uri"))('metadata -> "info:metadata"))

Let’s step through it line by line.

Read JSON records from an input file in HDFS

TextLine(args("input"))

This is done by using TextLine, a Scalding Source that views an input file as a collection of tuples of the form (offset, line). Here, offset is the offset within the file and line is the contents of an individual line from the file. This stream of tuples is called a pipe in Cascading terminology.

Flatten each JSON record into a tuple with fields corresponding to the song metadata extracted from the JSON record

.map('line ->
    ('songId, 'songName, 'albumName, 'artistName, 'genre, 'tempo,'duration)) { parseJson }

We now chain this to a map operation. Map-like functions operate over individual rows in a pipe, usually transforming them in some way. The new fields that are transformations of existing ones are added to the original tuple. In our code, we take a line and convert it into a named tuple of ('songId, 'songName, 'albumName, 'artistName, 'genre, 'tempo,'duration) using the parseJson UDF described above.

Transform the song id for each song into an entity id for the songs table

.map('songId -> 'entityId) { songId: String => EntityId(args("table-uri"))(songId) }

The next map transforms the song_id field from the tuple into an entity id, and adds this to the tuple in a field named entityId. EntityId in FijiExpress creates an EntityId for a given table URI out of the provided components, in this case, songId.

Pack song name, album name, artist name, genre, tempo, and duration for the song into an Avro record

.pack[SongMetadata](('songName, 'albumName, 'artistName, 'genre, 'tempo, 'duration)
      -> 'metadata)

Scalding lets you pack a set of fields into a single object using reflection. In our code, SongMetadata is an Avro record consisting of the fields described above. We now add a new field called metadata to each of the tuples whose value is the Avro record created from the song metadata. This will be the object that gets written to the Fiji table.

Write the Avro records to the column “info:metadata” in a row for the song in a Fiji table

.write(FijiOutput(args("table-uri"))('metadata -> "info:metadata"))

Finally, we use FijiOutput to create a Scalding Source that can write to a Fiji table. The argument table-uri contains the URI of the Fiji table to write to. We write the contents of the tuple field metadata (our SongMetdata Avro record) to the Fiji table column info:metadata. FijiExpress will automatically use the contents of the tuple field entityId as the entity id of the row in the Fiji table to write to.

Running the importers

Now we will run both the song metadata importer explained above and the song play history importer to import data into the Fiji tables songs and users.

FijiExpress programs or scripts can be run using the express command. First, we’ll demonstrate how to run the importers as precompiled jobs contained in a jar file. To run the song metadata importer, we use the following command.

express job --libjars "${MUSIC_EXPRESS_HOME}/lib/*" \
    ${MUSIC_EXPRESS_HOME}/lib/fiji-express-music-0.3.0.jar \
    org.fiji.express.music.SongMetadataImporter \
    --input express-tutorial/song-metadata.json \
    --table-uri ${FIJI}/songs --hdfs

After running the importer, you can verify that the Fiji table songs contains the imported data using the fiji scan command.

fiji scan ${FIJI}/songs --max-rows=5

You should see something like:

To run the song play history importer, we can use a similar command.

express job --libjars "${MUSIC_EXPRESS_HOME}/lib/*" \
    ${MUSIC_EXPRESS_HOME}/lib/fiji-express-music-0.3.0.jar \
    org.fiji.express.music.SongPlaysImporter \
    --input express-tutorial/song-plays.json \
    --table-uri ${FIJI}/users --hdfs

We can again use the fiji scan command to verify the importer was successful.

fiji scan ${FIJI}/users --max-rows=2 --max-versions=5

Both importers also exist in script form, in the folder ${MUSIC_EXPRESS_HOME}/scripts. As scripts, the importers can be run as:

express script --libjars "${MUSIC_EXPRESS_HOME}/lib/*" \
    ${MUSIC_EXPRESS_HOME}/scripts/SongMetadataImporter.express --hdfs

express script --libjars "${MUSIC_EXPRESS_HOME}/lib/*" \
    ${MUSIC_EXPRESS_HOME}/scripts/SongPlaysImporter.express --hdfs
